## Goals

- A stable, named, in-process Weave library API that lets a consumer say: **"here are payload bytes and a designator path (or payload IRI); record them"** — working-surface update and/or history versioning — with exactly the guarantees the CLI path gives (single-writer, fail-closed validation, snapshot verification, retry-safe no-op semantics), and **no subprocess spawn, no temp-file requirement on the caller**.
- The product-path law this serves (ruled, Stagecraft 2026-07-21): **consumer product paths call the API, never exec the CLI.** The standing counterexample: `weave` was absent from the consuming host's PATH until 2026-07-21 morning — CLI-spawn is a deployment coupling, not an API.

## Summary

Weave already versions payloads two-step: `weave payload update <file> <designatorPath>` replaces working bytes from a FILE source; `weave version --target designatorPath=...` records targeted working state into history via the version-plan machinery (`src/core/payload/update.ts`, `src/core/payload/version_intent.ts`, `src/core/weave/version_plan.ts`), with the CLI (`src/cli/run.ts`) owning invocation. `src/mod.ts` re-exports the core wholesale, so internals are *reachable* as a library today — but nothing is a *named, stable, documented* programmatic contract, and the file-source assumption in `payload update` forces callers to stage temp files.

This task was deferred on 2026-07-06 (Stagecraft disc "semantic-flow-in-the-platform" §6: no new API surface until a real trigger) and the wake fired 2026-07-21: the Stagecraft checkpoint press is the natural weave-version moment (publication cadence == version cadence; an unwoven pressed checkpoint is a second-class mesh state against SF-serialization doctrine), and the srd-rules publication estate now execs the CLI from its pipeline.

**Consumers, in ruled order:** (1) the `stagecraft/srd-rules` mesh estate's extraction/publication pipeline; (2) the Stagecraft checkpoint press — whose consumption is **explicitly not this task**: it folds into the Stagecraft press-completeness lane later, via the dependency-PIN idiom (full-commit pin + out-of-gate build + contract check). The deliverable here is the API itself, landed in weave.

## Discussion

- **Pass-bytes semantics vs the working-file capture contract.** The CLI's multi-target batch hashes the *working payload files* before batch capture and verifies after; a changed file refuses the whole batch. A bytes-in API changes what "the working surface" means at capture time: the API presumably (a) writes the passed bytes to the working payload file and (b) versions from that state — but whether (a)+(b) are one atomic composed call, two composable calls, or both-offered is the central spec question. The refusal/verification semantics must be restated precisely for the bytes-in path — nothing weakens.
- **The mesh stays file-backed.** Pass-bytes means the CALLER passes bytes; the mesh on disk remains the truth the API mutates. This is not an in-memory mesh.
- **Single-writer enforcement in-process.** The CLI's single-writer guarantee is process-shaped today. A library caller (long-lived daemon like the Stagecraft session authority, or a pipeline script) needs a stated concurrency contract: what serializes, what refuses, what happens on concurrent CLI use of the same mesh.
- **Return contract.** Callers need machine-usable results: the applied version-plan (created/updated files, new state segment, the payload artifact IRI, the current-history path), and typed refusals. `VersionPlan`/`PayloadVersionIntentPlan` are the natural seeds.
- **Behavior-spec law.** This is externally visible behavior spanning subsystems — per `wd.general-guidance`, an `sf.spec.*` note in semantic-flow-framework should ride the landing (proposed; spec review confirms scope).
- **Known 0.3.0 operational defects are SEPARATE work** (boarded on the Stagecraft side after the srd-rules phase-A run): `extract --all-terms` file-URL crash; extracted-term Knops missing `sflo:hasResourcePage` claims; whole-mesh `validate` heap exhaustion at ~6,900 files. Not in scope here unless the spec round proves one trivially adjacent — say so explicitly if so.

### r1 — adversarial spec review (2026-07-21)

All findings and verdicts in this review are PROPOSED. The PM adjudicates them on this note.

#### Ranked findings

1. **PROPOSED BLOCKER — The public operation and its bytes-capture boundary are not decidable.** The goal alternates among working-surface update, history versioning, and “update and/or history versioning,” while the discussion leaves one composed call, two calls, or both open. These are observably different contracts. The CLI exposes two separately committing commands: `payload update` invokes `executePayloadUpdate` (`src/cli/run.ts:1100-1154`), while `version` independently invokes `executeVersion` (`src/cli/run.ts:291-375`). The update writes the working file before it returns (`src/runtime/payload/update.ts:96-105`); `executeVersion` accepts no bytes and prepares from file-backed state (`src/runtime/weave/weave.ts:217-251`). The note does not say when a passed `Uint8Array` is copied, whether later caller mutation is visible, what disk state is compared to what input, when the working file becomes visible, whether a version refusal leaves the new working bytes installed, or whether a post-capture working-file change is ignored in favor of the passed/captured bytes. It also does not define whether “snapshot verification” means preserving the CLI's exact multi-file hash window or establishing a new compare-and-swap precondition for bytes-in. A builder would have to invent all of these rules. **REQUIRED RESOLUTION:** Pin the v1 operations and phase diagram before build: update-only, version-current, and/or update-and-version; call-admission copy/ownership of every byte buffer; discovery and validation; the exact pre/post verification set and comparison; the first possible write; what remains changed after each refusal/error stage; and the authoritative bytes after capture. Include one normative concurrent-mutation example. Do not use “exactly the guarantees” until each guarantee is scoped to an operation and phase.

2. **PROPOSED BLOCKER — “Single-writer guarantee/enforcement” contradicts landed policy, and concurrent API + CLI behavior cannot be supplied without expanding scope.** Weave does not currently enforce a single writer. The service guidance says the service must act as the single writer and explicitly says Weave provides no locking, rollback, journaling, or filesystem transactions (`documentation/notes/wu.cli-reference.weave.md:71-77`). The durable decision is even more direct: the requesting application owns serialization, locking, rollback, and retry orchestration (`documentation/notes/wd.decision-log.md:23-30`). The draft instead calls the CLI guarantee “process-shaped” and asks what the library serializes or refuses. An in-process mutex would serialize only cooperating calls and cannot refuse a concurrent CLI. A cross-process mesh lock would have to be shared by the CLI/runtime, contradicting “No CLI behavior change” (`wa.task...programmatic-version-api.md:51`). **REQUIRED RESOLUTION:** Choose one contract explicitly: (a) retain caller-owned single-writer preconditions, state that concurrent API/API and API/CLI mutation is outside the supported contract except for the bounded snapshot guard, and remove claims of enforcement; or (b) expand the task to a shared cross-process lock/protocol used by API and CLI, including acquisition identity, stale-lock recovery, lock scope, refusal type, and compatibility behavior. The current wording cannot support both.

3. **PROPOSED BLOCKER — The binary promise is not a wrapper-sized change and currently has no CLI-equivalent path.** Core payload update accepts `replacementPayloadTurtle: string`, rejects blank text, and emits `PlannedFile` (`src/core/payload/update.ts:11-28`, `src/core/payload/update.ts:137-145`). Runtime update reads and writes text (`src/runtime/payload/update.ts:226-259`, `src/runtime/payload/update.ts:407-459`) and the settled-shape assertion requires the payload to be an `sflo:RdfDocument` with `_history001` (`src/core/payload/update.ts:165-193`). `PlannedBinaryFile` only establishes a binary *created-output* shape (`src/core/planned_file.ts:1-9`, `src/core/weave/version_plan.ts:3-9`). First payload versioning correctly emits a binary historical snapshot (`src/core/weave/weave.ts:1175-1237`), but later payload versioning always puts decoded `currentPayloadTurtle` in `createdFiles` (`src/core/weave/weave.ts:1682-1760`), even though the loader separately carries `currentPayloadBytes` for non-text extensions (`src/runtime/weave/artifact_loaders.ts:71-95`). Therefore “binary round-trip” and CLI/API equivalence cannot be satisfied by carving an API around the landed update/version seams; later binary advancement is directly adjacent correctness work. **REQUIRED RESOLUTION:** Either include end-to-end byte-preserving working update plus first/later history advancement in this task (and acknowledge the necessary shared runtime/CLI behavior correction), or narrow v1 explicitly to a defined UTF-8 text/RDF class and remove the binary/`Uint8Array` promise. If binary remains, specify zero-length policy, exact byte identity, input-buffer copying, extension/content-kind classification, invalid UTF-8 behavior for text/RDF paths, and first and later no-op comparisons.

4. **PROPOSED BLOCKER — Retry-safe no-op and snapshot verification are overgeneralized beyond their landed batch domain.** The runtime attempts the special payload batch only when there are more than one exact targets and overwrite is false (`src/runtime/weave/version_execution.ts:625-631`). Only that path hashes inputs and reloads settled payload candidates (`src/runtime/weave/version_execution.ts:136-201`). Batch planning skips already-current members by exact byte/text and requested-segment checks (`src/core/weave/weave.ts:574-580`, `src/core/weave/weave.ts:907-960`). A settled single exact target is not loaded as weaveable in the normal path (`src/runtime/weave/candidate_loader.ts:311-367`) and uncovered explicit targets refuse (`src/runtime/weave/candidate_loader.ts:38-64`); it does not inherit the batch no-op rule. The note's goal and “single and batch” equivalence family do not expose this distinction. **REQUIRED RESOLUTION:** Decide whether an API request containing one payload is a one-member coherent batch with snapshot verification and success/no-op, or mirrors current single-target `executeVersion` refusal behavior. Pin overwrite behavior, empty batch behavior, duplicate/aliased targets, and whether naming-only requests intentionally create a state rather than no-op. If retry-safe no-op is a stable API guarantee, make it an explicit per-input result state and test one-member as well as multi-member retries.

5. **PROPOSED HIGH — “Whole-batch refusal” is being conflated with transactionality across working updates and output writes.** Landed batch behavior validates a `VersionPlan` before version writes, but the durable decision calls it “not a transactional filesystem boundary” (`documentation/notes/wd.decision-log.md:23-30`). `writePreparedVersion` preflights paths/RDF, then writes text creates, binary creates, and updates in separate loops (`src/runtime/weave/version_execution.ts:868-933`, `src/runtime/weave/version_execution.ts:997-1021`), so an I/O failure can leave partial output. With the proposed pair, one or more working updates can also commit before version planning refuses. The CLI docs' retry statement does not define recovery from every possible mid-target partial write. **REQUIRED RESOLUTION:** Define three separate guarantees: whole-request semantic planning/refusal before *version-output* writes; working-surface mutation behavior; and non-transactional I/O failure behavior. State whether the composed API plans from an in-memory byte overlay before changing working files, and whether a validation refusal preserves old working bytes or installs the requested bytes. Add write-failure injection cases at working-update, text-create, binary-create, and support-update phases; do not label the filesystem result “all-or-nothing” unless a real transaction/rollback mechanism is added.

6. **PROPOSED HIGH — Target identity, mesh identity, and source ownership are unnamed.** Existing public requests accept a designator path, not a payload IRI (`src/core/targeting.ts:6-15`, `src/core/weave/requests.ts:16-19`), while the goal promises either. The note gives no `meshRoot` type/default, no path-versus-IRI discriminated union, no rule for both values, no requirement that an IRI be under the loaded mesh base, no percent-encoding/root-designator rule, and no duplicate identity rule after normalization. More seriously, runtime version loading prioritizes a repository-source floating locator over the mesh-relative working path (`src/runtime/weave/artifact_loaders.ts:76-86`), while payload update writes the mesh-relative working file under the workspace (`src/runtime/payload/update.ts:330-367`). A naïve composed API can therefore update one file and version another. **REQUIRED RESOLUTION:** Pin the request identity schema and normalization/refusal rules, including `meshRoot`; designator path versus payload IRI; root; percent encoding; outside-mesh IRIs; both/neither; duplicate aliases; exact-only versus recursive targets; and payload-versus-non-payload identity. Rule repository-backed/floating-source payloads explicitly: update the resolved source under the existing local-access policy, materialize a mesh-local working file and change inventory, or refuse with a typed unsupported-source error. This ruling must be validated against the first srd-rules consumer shape rather than assumed.

7. **PROPOSED HIGH — The “stable public surface” has neither exact symbols nor a stability boundary.** The draft correctly observes wholesale reachability: root `src/mod.ts` re-exports core, runtime, and CLI (`src/mod.ts:1-3`), whose module barrels recursively export payload and weave symbols (`src/core/mod.ts:1-10`, `src/runtime/mod.ts:1-10`). `executeVersion` is already a named reachable function; what is missing is a deliberately supported bytes-in contract. “`src/api/` (or `core/api/`)” leaves architecture to the builder, but `core/api/` would violate the repository law that core avoids filesystem/runtime assumptions (`documentation/notes/wd.codebase-overview.md:15-19`). No function/type/error names, import example, experimental marker, or exact-versus-latitude mark is provided. **REQUIRED RESOLUTION:** Rule `src/api/` (not `core/api/`) for the filesystem-owning façade; pin the v1 function, request, result, target-result, and error export names and their root import path; state which are stable and which existing wholesale exports remain incidental; and include a compiling normative import/call example. If the PM deliberately leaves naming latitude to build, mark every such identifier LATITUDE and require adjudication before merge, not silent invention.

8. **PROPOSED HIGH — The promised result and typed refusal set are not derivable from the proposed “natural seeds.”** `VersionPlan` carries mesh base, versioned designator paths, and planned file contents, but no per-target payload IRI, chosen history/state/manifestation, current-history path, applied/no-op status, or refusal (`src/core/weave/version_plan.ts:3-9`). `PayloadVersionIntentPlan` has payload IRI/current history and optional next-state hint, but it describes the separate inventory-intent operations and not an applied version (`src/core/payload/version_intent.ts:33-40`, `src/core/payload/version_intent.ts:49-145`). Runtime `VersionResult` exposes paths only (`src/runtime/weave/weave.ts:94-99`). Existing failures span input, runtime, config, path-access, snapshot-conflict, create-collision, and raw I/O domains; the CLI reduces them to message text (`src/cli/run.ts:1967-1993`). The note leaves thrown errors versus result unions open and gives no stable machine code or partial-write representation. **REQUIRED RESOLUTION:** Specify a per-target success algebra (`applied`/`alreadyCurrent`, resolved identity, selected history/state/manifestation and snapshot path), batch-created/updated working and version paths, and whether the full mutable-content `VersionPlan` is exposed. Specify stable refusal/error codes and fields at minimum for invalid request/identity, target not found/not payload, malformed mesh/inventory/config, inconsistent batch policy, unsupported source/content, snapshot conflict, validation/plan conflict, path/permission failure, and write failure with phase/possible partial paths. Rule throw-versus-return and make message text diagnostic, not the machine contract.

9. **PROPOSED HIGH — Configuration/default precedence is described as “mirror the CLI” but is not restatable from the note.** CLI-level general segment flags are copied onto targets only when the target omitted that field (`src/cli/run.ts:1703-1777`). Runtime then loads mesh and ancestor/target-scoped effective config (`src/runtime/weave/execution_config.ts:70-138`, `src/runtime/weave/execution_config.ts:205-245`), and a batch refuses inconsistent target-scoped planning policies (`src/runtime/weave/version_execution.ts:734-780`). Layout resolution uses explicit target history over current intent/default policy and explicit state over `hasNextStateSegmentHint`/policy (`src/core/weave/payload_version_layout.ts:45-97`, `src/core/weave/payload_version_layout.ts:100-240`). `historyTrackingPolicyOverride` and `overwriteExistingState` are also existing version controls (`src/runtime/weave/weave.ts:48-54`, `src/core/weave/requests.ts:16-19`) but are not integrated into the proposed options contract. Explicitly preserve or strengthen the current exclusion of config/support files from the capture hash; do not imply they are verified today.

10. **PROPOSED HIGH — “Fail-closed validation” is not scoped to an actual validation operation.** `executeVersion` normalizes, prepares, and validates the planned RDF files before writing (`src/runtime/weave/weave.ts:217-251`, `src/runtime/weave/version_execution.ts:936-994`); it does not run whole-mesh validation. #### Open-issue lean verdicts

- **API shape — REFUTE the composable pair as the primary v1 contract; ENDORSE a composed coherent-batch call as primary.** The pair reproduces the CLI deployment seam inside one process but cannot by itself give bytes-in batch validation, capture, or refusal semantics across the update/version gap. A lower-level `updatePayloadBytes` and version-current call may exist for human-equivalent workflows only if their weaker boundary is explicit. The first consumer needs one request that accepts all bytes/identities, captures them, plans the coherent batch, and returns per-target applied/no-op outcomes. The thin convenience should not be the only call that owns the strongest guarantees.
- **Batch surface — ENDORSE multi-target v1, with conditions.** The first consumer is inherently multi-target and the landed deterministic canonical ordering/merged support progression is the correct semantic base. Define the API request as a coherent batch even at cardinality one, reject duplicates after identity normalization, support exact payload targets only, and call the guarantee “whole-plan pre-write refusal,” not filesystem transactionality. Decide separately whether update-only accepts batches.
- **Naming/segments — ENDORSE one options object only after an explicit precedence table is added.** Use batch defaults plus per-item overrides analogous to CLI general flags and target fields, but also include persisted intent, target-scoped config, built-in policy, `historyTrackingPolicyOverride`, and overwrite behavior. “Mirror exactly” is not an executable rule by itself.
- **Error taxonomy — ENDORSE typed thrown errors, not message matching, based on the existing runtime convention.** Prefer one stable public error base with discriminated `code`, `stage`, optional target/path/batch details, and `cause`; subclasses are optional. Applied/no-op is a normal result, not an error. Keep operational I/O failure distinct from semantic refusal and disclose possible partial paths for write-stage failures.
- **Stable-surface boundary — ENDORSE dedicated `src/api/`; REFUTE `core/api/`.** The façade must perform filesystem/config/runtime orchestration and therefore does not belong in pure core. Add explicit API exports and document only those names as stable even while older wholesale exports remain reachable. Pin exact names/imports before merge.
- **Binary payloads — ENDORSE `Uint8Array` as the only payload-byte representation, conditional on owning the end-to-end fix.** Copy inputs at admission and preserve exact bytes.
