# claude-opus-4.7_2026-05-23_0839_15

Strong task overall — the slicing, scope discipline, and vocabulary distinctions (`Region` vs `Panel` vs rendered "section") are the parts I'd most want to keep. Here's the feedback worth your time, roughly in order of how much it could bite you later.

## Things that may bite during implementation

**1. Panel data requirements vs. targeting are conflated.** The example shows both `sfcfg:hasPanelTargetClass sflo:DigitalArtifact` and `sfcfg:hasPanelDataRequirement sfcfg:panelDataRequirement_currentWorkingSource` on the same selection. Are these AND-ed? Is one a "should this panel apply" gate and the other a "should the runtime bother computing data" hint? At first glance they overlap (a class targets shape, a data requirement targets reality), and you'll want a clear semantics doc before the second slice or you'll be retrofitting later.

**2. Inheritance/merge semantics are deferred to Open Issues — but they need at least a stub answer by slice 2.** Application → mesh → Knop layering with additive panel selections and nearest-wins templates sounds fine until two layers contribute panel selections with the same `panelOrder` or the same underlying `ResourcePagePanel`. The current draft has no replace/remove vocabulary and no dedup rule. Naming the rule even tentatively ("ties broken by mesh-closer-than-application; same panel referenced twice keeps the nearer occurrence") would prevent the first real conflict from surprising you.

**3. Unknown/missing panel identity policy is unspecified.** Slice 2 starts referencing panels by stable identity (`sfcfg:resourcePagePanel_children`). What if a future build removes one, or a mesh references an identity Weave doesn't recognize? Silent drop vs. diagnostic vs. hard error matters and should be decided before defaults ship in `defaults/application.ttl`.

**4. Custom-page opt-in mechanism is named but not designed.** The Decisions and Plan both mention custom pages opting into default panels "explicitly," with authored regions ordering before generated panels when no order is given. But the mechanism — a presentation-config reference? a flag on `ResourcePageDefinition`? a panel-list property? — is left implicit. This is the slice-3 interface and worth a placeholder sketch in Open Issues.

## Smaller observations

**5. "Selection" is doing two jobs.** `sfcfg:hasResourcePagePanelSelection` collects selections; each selection element is also a `ResourcePagePanelSelection`. Singular and plural meanings on the same noun. `PanelEntry` for the element or `PanelBinding` would read better, but flag for ontology review rather than fix now.

**6. Hash IRIs are good for dereferencing, but reuse mechanics aren't addressed.** `#children-panel` is fine inside one config, but if a mesh wants to override just that selection, does it reach in by hash IRI or replace the whole `PresentationConfig`? If the former, the inheritance rules above need to include "override by IRI match."

**7. Zero-drift acceptance gate could be sharper.** "Preserve current generated HTML output" is the bar for slice 1, but the Testing section only mentions running fixture-stable comparisons. Worth committing to a baseline snapshot taken pre-refactor as the literal accept gate — otherwise "no drift" is judged by whichever tests happen to exist.

**8. Inline CSS → `ResourcePageStylesheet` migration path isn't called out.** Decision 11 (treat built-in stylesheet as embedded code with stable RDF identity, materialize later) is fine but skips the intermediate state. When does the RDF identity show up? Slice 2 alongside the panel identities? Worth being explicit.

**9. Children / large-mesh ergonomics aren't mentioned.** A `children` panel on a deep node could be huge. Whether panels can declare display limits or pagination policy is probably out of scope for slice 1, but worth a one-line Non-Goal so it doesn't get smuggled in.

**10. The "synthesized default document model" idea is scattered.** It appears in Goals, Summary, Discussion, Decisions, and Non-Goals. It's central enough to deserve a single named subsection — partly because the diagnostics decision ("don't call it a `ResourcePageDefinition`") hinges on it.

## What I'd leave alone

- The slice ordering (internal extraction first, ontology second, custom-page sharing third). Tempting alternative: paper-design ontology first. But you'd over-commit to terms before knowing the panel shape, and the existing Decisions already constrain the internal model enough that the RDF mapping won't be a surprise.
- The deferral of raw authored HTML and third-party panel plugins. Both have safety/trust dimensions that don't belong in a first slice.
- The decision to keep runtime owning model assembly (filesystem, config resolution, access policy) and core owning model *types*. That split is the right one.

Net: this is ready to start slice 1 from. The four items at the top are the ones I'd write a paragraph on before slice 2 begins, not before slice 1.
